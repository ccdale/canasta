# 04 — AI Player Behavior and Seed Effects

## What the AI actually is in this project

The AI player is a rule-following bot layer that sits on top of the same engine API used by humans.
It does not bypass rules and it does not inspect hidden state.

On a bot-controlled turn, the flow is:

1. Draw from stock.
2. Repeatedly attempt to create melds from current hand.
3. Choose one legal discard.

This is implemented by `play_bot_turn` in `src/canasta/bots.py`.

---

## Where bot logic lives

Two modules split responsibilities:

| Module | Responsibility |
|--------|----------------|
| `src/canasta/bot_strategies.py` | Decision heuristics (`choose_meld_indexes`, `choose_discard_index`) |
| `src/canasta/bots.py` | Factory (`build_bot`) and turn orchestration (`play_bot_turn`) |

The strategy classes all implement the `TurnBot` protocol:

- `RandomBot`
- `GreedyBot`
- `SafeBot`
- `AggroBot`
- `PlannerBot`

---

## Shared decision constraints

All bots are constrained by the same helper functions and rules:

- Meld candidates are generated by `_eligible_natural_meld_candidates`.
- Non-opening candidates are natural-only groups of same-rank cards, size 3+.
- Opening candidates also include legal split-rank natural combinations (multiple ranks in one action).
- If opening is required, candidates below `OPENING_MELD_MINIMUM` are excluded.
- Discard choices must pass `can_discard`, so illegal discards (for example red threes) are never selected.

This means differences between bots are mostly ranking preferences, not rule interpretation.

---

## Strategy behavior by bot

### Random

- Chooses a random eligible meld candidate.
- Chooses a random legal discard index.
- Uses its own random number generator object.

### Greedy

- Meld: maximize immediate meld score, then meld length.
- Discard: minimize discarded card points, preferring non-wild ties.

### Safe

- Meld: if already opened, often holds cards and skips new melding.
- Opening meld: picks the lowest-value legal opener that still meets threshold.
- Discard: defensive preference for black threes (freeze pressure), avoids wilds, and tends to shed singletons.

### Aggro

- Meld: prioritize largest meld size, then meld points.
- Discard: dump higher-point cards sooner.

### Planner

- Meld: maximize meld points, then size, then natural density.
- Discard: tries to preserve wilds and grouped ranks for future meld flexibility.

## What seed controls (and what it does not)

There are two independent seed domains in this project:

1. Engine seed (`CanastaEngine(seed=...)`): controls deck shuffle and therefore deal/draw order.
2. Bot seed (`--bot-seed`): controls bot randomness only, and only where a bot uses randomness.

In CLI and GUI seat-controller setup, the base bot seed is offset per seat:

- North bot gets `bot_seed + 1`
- South bot gets `bot_seed + 2`

So even with the same `--bot-seed`, north and south do not share the same RNG stream.

Important: currently only `RandomBot` consumes RNG directly.
`GreedyBot`, `SafeBot`, `AggroBot`, and `PlannerBot` are deterministic given the same game state.

---

## Why changing seed changes outcomes

If `RandomBot` is active, a different seed changes which random legal choices are made:

- which meld candidate is chosen when multiple are legal
- which discard is chosen among legal options

Those choices alter future hand shape and discard pile state, which can cascade into very different games.

If no random bot is active, changing `--bot-seed` alone will not change decisions, because non-random strategies do not sample RNG.

---

## Reproducibility tips

To reproduce bot behavior reliably, lock all of the following:

1. Same code version.
2. Same initial engine state (or same engine seed when constructing engine).
3. Same bot seat assignment.
4. Same `--bot-seed`.

For bot-vs-bot experiments, record both seat strategies and seed.
Because seat offsets differ (`+1`, `+2`), swapping north/south changes behavior even with the same base seed.

---

## Practical examples

CLI:

```bash
uv run canasta --north random --south greedy --bot-seed 7
uv run canasta --north random --south greedy --bot-seed 8
```

Only the north random bot should vary due to bot-seed changes.

GUI:

```bash
uv run canasta-gui --north aggro --south planner --bot-seed 7
```

In this pairing, strategy decisions are deterministic from state, so bot-seed does not directly affect choice selection.

## Cross-checks in tests

`tests/test_bots.py` verifies:

- bot factory wiring for all five strategies
- random bot can complete legal turns
- heuristic tendencies for safe/aggro/planner behavior

Engine tests use fixed seeds for deterministic setup where required.
